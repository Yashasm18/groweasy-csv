# MASTER BLUEPRINT — GrowEasy AI CSV Importer
### Build & Delivery Handoff Document

*A single, self-contained specification for a **new repository** whose purpose is to build an AI-powered CSV importer: it accepts **any** valid CRM lead CSV (Facebook export, Google Ads, Excel, real-estate CRM dumps — no fixed columns), maps every row into a fixed 15-field CRM schema using an LLM, applies deterministic validation, and shows the result through a 4-step web UI. The output is a **publicly hosted app + public GitHub repo + README**, submitted for the GrowEasy Software Developer assignment.*

*This document is written to be handed to a capable coding agent (Gemini) and worked through **top-to-bottom, one full phase per session**. It is deliberately explicit and over-specified: assume the builder has no outside context beyond these docs.*

# PART I — ENGINEERING GROUND RULES

Non-negotiable standards for every line of code. Re-read before each phase. These are what the "Code Quality" and "Overall Engineering" evaluation criteria grade.

## 1. Modularity — one module, one job
- Backend responsibilities live in separate modules: **CSV parsing**, **LLM client**, **batch orchestration/extraction**, **validation/post-processing**, **routing**, **config**. No God-file.
- **Program to an interface for the LLM.** Define an `LlmProvider` interface (Part III). Only `services/llm/geminiProvider.ts` imports the Gemini SDK. Nothing else in the codebase imports it. This is what lets you swap Gemini → Claude/OpenAI by editing one file if the key fails (see `docs/ROADBLOCKS.md` §B).
- **Dependency injection.** Pass collaborators in (the provider, the config); do not `new` them deep inside business logic or reach for globals.
- **Pure functions for validation.** The validator takes a raw candidate record and returns a clean record or a skip reason — same input, same output, no side effects. This is what makes it unit-testable.

## 2. No hard-coding
- **All tunables live in one config module** (`backend/src/config.ts`), read from environment with sane defaults: batch size, model name, concurrency, retry counts, request size limit, allowed CORS origin, port. No magic numbers scattered in code.
- **Secrets only from environment variables.** `GEMINI_API_KEY` is read from `process.env`, never committed, never logged, never sent to the frontend. There is a `.env.example` with placeholder values; the real `.env` is git-ignored.
- **The 15 fields and the two enums are defined once** in `backend/src/domain/crm.ts` (and mirrored in a frontend type). Every other file imports them. Changing the schema is a one-file edit.
- **OS-portable code.** Use `path.join`, forward-slash imports; never hardcode `\` or `/` separators or absolute machine paths.

## 4. No temporary patch fixes
Solve the underlying problem, not the symptom. No `// TODO: fix later` in merged work, no commented-out dead code, no `any`-casting to silence the type-checker. If the LLM occasionally returns a bad value, fix it in the validator, don't paper over one case. If a design strains, refactor it.

## 5. Comments & types
Comment the **why**, not the **what**. A docstring on every exported function (purpose, params, returns, and any invariant — especially "this function is the last line of defense: the LLM output is untrusted here"). **Static typing everywhere** — `strict: true` in both `tsconfig.json` files; no implicit `any`; the build must type-check clean.

## 6. Git best practices
- **Branch per phase:** `feat/p2-extraction`, `feat/p3-frontend`. Never commit straight to `main` while building; open a PR per phase (or at minimum, tag each gate).
- **Small, atomic, conventional commits** (`feat:`, `fix:`, `test:`, `refactor:`, `docs:`, `chore:`). Each commit leaves the tree working and type-checking.
- **Never commit** `.env`, API keys, `node_modules`, `dist`, `.next`, or any real lead data beyond the synthetic samples.
- **No AI/tool/authorship attribution** in commit messages, PR bodies, tags, comments, or code. History reads as the operator's own work, end to end.
- Type-check and lint **before** committing.

## 7. Professional standards
Clean readable code, clear names, small functions, formatter + linter (ESLint + Prettier) configured and passing. **Fail loudly at boundaries:** validate the uploaded file (type, size, non-empty) and raise clear, specific errors the UI can show. **Determinism where it matters:** the validator is fully deterministic; only the LLM step is probabilistic, and its output is always re-checked.

## 8. Proper logging
A tiny structured logger in `util/logger.ts` (levels: debug/info/warn/error), one line per import request with a correlation id, per-batch timing, and the parsed/skipped counts. **Never log the API key or full lead PII** — log counts and row indices, not raw emails/phones. Timestamps in ISO.

## Project-specific inviolable rules
These override convenience and "just make it work":

1. **The LLM is untrusted; code has the final say.** Every field the model returns passes through `validator.ts`: `crm_status` and `data_source` are clamped to the allowed enum (blank if no match), `created_at` is verified with `new Date()`, emails/phones are de-duplicated and overflow pushed to `crm_note`, newlines are escaped to `\n`. The model **never** decides the final shape unchecked. (Part III §Validation.)
2. **The skip rule is absolute.** A record with **neither** a usable email **nor** a usable mobile number is skipped — always, enforced in code, and reported in the "skipped" list with the reason. Never emit a lead that violates this.
3. **The API key lives only on the backend.** The frontend never sees it. All LLM calls go through the Express backend. A build that calls Gemini from the browser fails Inviolable Rule 1 of security and is an automatic assignment red flag.
4. **Any CSV, no fixed columns.** Never assume column names or order. The parser reads whatever headers exist; the LLM does the semantic mapping; the validator enforces the schema. A hardcoded `row["Email"]` lookup anywhere is a bug.
5. **Prove on the four sample CSVs before claiming done.** `docs/sample-data/` contains a Facebook export, a Google Ads export, a real-estate CRM dump, and a deliberately messy edge-case file. Every backend phase is checked against all four. They are your acceptance corpus.
6. **Ship a working, hosted app over a feature-rich broken one.** With a tight deadline, a deployed app that does the core loop correctly beats a local app with every bonus. Follow the phase order; treat bonuses (Part V) as strictly optional polish after the core gate.
7. **Ground every decision in these docs.** Part III is self-contained and sufficient to build every phase. The deep dives are expanded reference. If these docs are genuinely silent or self-contradictory on a decision, STOP and surface it in `docs/PROGRESS.md` under "Open decisions" with options — do not guess and do not invent requirements the assignment did not state.

# PART II — SYSTEM OVERVIEW & LOCKED DECISIONS

## What we are building
A two-part web application:
- **Frontend (Next.js):** a 4-step wizard — Upload → Preview → Confirm → Results.
- **Backend (Node + Express):** receives the CSV, parses it, sends rows to the LLM in **batches** for intelligent field mapping, validates and cleans the results deterministically, and returns structured JSON (parsed records + skipped records + totals).

The intelligence is **LLM-led, code-guarded**: the LLM handles the semantic problem (mapping arbitrary columns and classifying free text into the schema); deterministic code handles everything code does better (validation, enum enforcement, de-duplication, date checks, the skip rule).

```
 CSV FILE → [parse]→ raw rows → [batch → LLM map to schema] → candidate records → [validate/clean] → { records, skipped, summary } → UI
   upload    PapaParse            Gemini (JSON schema)          untrusted            enums, skip rule,        4-step wizard
                                                                                     dedup, dates, \n
```

## The assignment, restated precisely (source of truth for requirements)

| # | Field key | Meaning | Notes / rules |
|---|---|---|---|
| 1 | `created_at` | Lead creation date | Must be parseable by JS `new Date(value)` (not `Invalid Date`). If the source has no date, leave `""`. |
| 2 | `name` | Lead full name | |
| 3 | `email` | Primary email | If a cell has multiple emails, use the **first**; append the rest to `crm_note`. |
| 4 | `country_code` | Phone country code | e.g. `91`, `1`. Split from the raw phone if embedded (see Validation). |
| 5 | `mobile_without_country_code` | Mobile number, no country code | Digits only, country code stripped. If multiple numbers, first here, rest to `crm_note`. |
| 6 | `company` | Company name | |
| 7 | `city` | City | |
| 8 | `state` | State / region | |
| 9 | `country` | Country | |
| 10 | `lead_owner` | Assigned owner / agent | |
| 11 | `crm_status` | Lead status | **Exactly one of the allowed enum** or `""`. See below. |
| 12 | `crm_note` | Consolidated notes | Remarks, follow-up notes, **extra** emails/phones, and any leftover/miscellaneous columns land here. |
| 13 | `data_source` | Lead source | **Exactly one of the allowed enum** or `""` (blank if no confident match). See below. |
| 14 | `possession_time` | Property possession timeframe | Real-estate specific; else `""`. |
| 15 | `description` | Additional free-text description | |

**Every output record has all 15 keys.** Missing information → empty string `""`, never `null`/`undefined`/omitted. This keeps the CSV/table rectangular.

### Allowed `crm_status` values (choose exactly one, or `""`)
```
GOOD_LEAD_FOLLOW_UP
DID_NOT_CONNECT
BAD_LEAD
SALE_DONE
```

### Allowed `data_source` values (choose exactly one, or `""` if unsure)
```
leads_on_demand
meridian_tower
eden_park
varah_swamy
sarjapur_plots
```
> `data_source` is a **project-name** enum — the LLM must map free text to one of these only when confident (e.g. a column or value clearly references "Eden Park"). **When in doubt, leave it blank.** A wrong confident guess is worse than a blank.

### The absolute rules
1. **Skip rule:** if a record has neither a usable email nor a usable mobile → **skip it**, record it in `skipped` with `reason: "no email or mobile"`.
2. **Multiple emails / phones:** first value wins the field; remaining values are appended to `crm_note` (e.g. `Other emails: a@x.com, b@y.com`).
3. **`created_at`:** must satisfy `!isNaN(new Date(value).getTime())`; otherwise blank it.
4. **CSV validity of output:** any newline inside a field value is escaped to the two-character string `\n` (so each record is one clean CSV row / one clean JSON string).
5. **Enums:** `crm_status` and `data_source` are clamped to the allowed set in code; anything else becomes `""`.

### The 4-step frontend workflow
- **Step 1 — Upload:** drag-and-drop zone **and** a file picker. Accept `.csv` (and gracefully handle `text/csv`, `application/vnd.ms-excel` mislabels). Reject non-CSV / empty with a clear message.
- **Step 2 — Preview:** parse the CSV **client-side** (PapaParse) and show the raw rows in a responsive table with **sticky header** and **horizontal + vertical scroll**. **No AI call yet.** Show row/column counts. A "Confirm Import" button advances.
- **Step 3 — Confirm:** clicking Confirm sends the file to the backend `POST /api/import`; show a **loading / progress** state while the AI processes (criterion: loading states are graded).
- **Step 4 — Results:** show the parsed records in one table and the skipped records in another (or a tab), plus totals (`total`, `parsed`, `skipped`). Same sticky-header/scroll treatment. Offer "Import another file" to reset.

## Locked decisions (do not re-litigate mid-build)
| Decision | Choice | Why |
|---|---|---|
| Frontend | **Next.js (App Router) + TypeScript + Tailwind CSS** | Assignment names Next.js; Tailwind gives fast, responsive, dark-mode-ready UI. |
| Backend | **Node.js + Express + TypeScript, separate service** | Assignment explicitly names Node/Express; "Backend Quality / API design" is a graded criterion, so a real Express service (not just Next API routes) is the intended shape. |
| LLM | **Google Gemini** via the LLM interface | Operator's choice; generous free tier. Behind `LlmProvider` so it's swappable. |
| Gemini model | **`gemini-2.5-flash`** (fast, cheap, strong at structured extraction) | Flash tier is the right speed/cost point for batch extraction. Verify the exact current model id at build time; keep it in config. |
| Structured output | Gemini **`responseMimeType: "application/json"` + `responseSchema`** | Forces schema-valid JSON; eliminates fragile prose-parsing. |
| CSV parsing | **PapaParse** (client for preview, and server via `papaparse`) | Battle-tested; handles quotes, embedded commas/newlines, delimiter sniffing, BOM. Never hand-split on commas. |
| Upload transport | **`multipart/form-data`** file upload (Multer) to the backend | Backend is the authoritative parser (source of truth), so it receives the raw file, not pre-parsed rows. |
| Batch size | **20 rows/LLM call** (config `BATCH_SIZE`, default 20) | Balances token cost, latency, and rate limits. Tunable. |
| Concurrency | **2 batches in flight** (config `LLM_CONCURRENCY`, default 2) | Respects free-tier RPM while staying fast. |
| Retry | **Up to 3 attempts/batch, exponential backoff** on rate-limit/transient/parse errors | Bonus "retry mechanism"; also basic robustness. |
| Progress | **Server-Sent Events (SSE)** streaming per-batch results (Phase 4) | Bonus "progress indicators"; also better UX than one long spinner. Plain JSON response is the Phase-2/3 baseline. |
| Deploy | **Frontend → Vercel; Backend → Render (web service)** | Both have usable free tiers; see deep dive 03 for the cold-start caveat. |
| Language/tooling | TypeScript, ESLint + Prettier, both apps | Type safety is a graded criterion. |

# PART III — TECHNICAL REFERENCE

*Self-contained: enough to build every phase from. Full expansion in `docs/deep_dives/` and `docs/PROMPT_ENGINEERING.md`.*

## The domain model (`backend/src/domain/crm.ts`)
```ts
export const CRM_FIELDS = [
  "created_at", "name", "email", "country_code", "mobile_without_country_code",
  "company", "city", "state", "country", "lead_owner", "crm_status",
  "crm_note", "data_source", "possession_time", "description",
] as const;
export type CrmField = (typeof CRM_FIELDS)[number];

export const CRM_STATUS_VALUES = [
  "GOOD_LEAD_FOLLOW_UP", "DID_NOT_CONNECT", "BAD_LEAD", "SALE_DONE",
] as const;
export type CrmStatus = (typeof CRM_STATUS_VALUES)[number] | "";

export const DATA_SOURCE_VALUES = [
  "leads_on_demand", "meridian_tower", "eden_park", "varah_swamy", "sarjapur_plots",
] as const;
export type DataSource = (typeof DATA_SOURCE_VALUES)[number] | "";

// Every field is a string; missing info is "" (never null/undefined).
export type CrmLead = Record<CrmField, string>;

export interface SkippedRow { rowIndex: number; reason: string; raw: Record<string, string>; }
export interface ImportSummary { total: number; parsed: number; skipped: number; }
export interface ImportResult { summary: ImportSummary; records: CrmLead[]; skipped: SkippedRow[]; }
```

## The LLM interface (`services/llm/LlmProvider.ts`) — the swap seam
```ts
export interface LlmProvider {
  /** Map a batch of raw CSV rows to candidate CRM records. MUST return one
   *  candidate per input row, in the same order (echo rowIndex). Output is
   *  UNTRUSTED and re-validated downstream. Throws on transient/API errors so
   *  the extractor's retry can catch them. */
  mapBatch(rows: RawRow[]): Promise<CandidateRecord[]>;
}
export interface RawRow { rowIndex: number; cells: Record<string, string>; }  // cells keyed by original header
export interface CandidateRecord { rowIndex: number; fields: Partial<CrmLead>; }
```
Only `geminiProvider.ts` imports the Gemini SDK. To swap providers (see `docs/ROADBLOCKS.md` §B) you write one new file implementing this interface and change one line in `config.ts`/wiring.

## The extraction pipeline (`services/extract/extractor.ts`)
Orchestration only — no SDK, no HTTP:
1. Receive normalized `RawRow[]` from the parser.
2. Chunk into batches of `BATCH_SIZE`.
3. Run batches with concurrency `LLM_CONCURRENCY` (a simple promise pool).
4. For each batch call `provider.mapBatch(batch)` inside a **retry wrapper** (up to `LLM_MAX_RETRIES`, exponential backoff with jitter; retry on rate-limit/5xx/timeout/JSON-parse failures).
5. If a batch **permanently** fails after retries, **degrade gracefully**: fall back to a deterministic best-effort mapping for those rows (regex email/phone extraction + copy-through of obvious columns) rather than dropping them — and mark them so the UI/log can note reduced confidence. Never let one bad batch fail the whole import.
6. Pass every `CandidateRecord` through `validator.ts`.
7. Assemble `ImportResult`.

## Validation & cleaning (`services/validate/validator.ts`) — the code that has the final say
Pure, deterministic, unit-tested. For each candidate record:
- **Normalize phones:** strip spaces/dashes/parens. If a `+CC`/`00CC`/leading-country pattern is present, split into `country_code` (digits, no `+`) and `mobile_without_country_code` (the national number, leading zero removed). Collect **all** phone-like tokens found across the row; first → the field, rest → `crm_note`.
- **Normalize emails:** collect all `\S+@\S+\.\S+` tokens; first → `email`, rest → `crm_note`.
- **Enforce skip rule:** if after the above there is **no** email **and** no mobile → return a skip with `reason: "no email or mobile"`.
- **Clamp enums:** if `crm_status` ∉ `CRM_STATUS_VALUES` → `""`. If `data_source` ∉ `DATA_SOURCE_VALUES` → `""`. (Trim/upper-case/underscore-normalize before comparing so `"Good lead follow up"` maps to `GOOD_LEAD_FOLLOW_UP`.)
- **Validate `created_at`:** if `isNaN(new Date(v).getTime())` → `""`. (Optionally normalize to ISO; keep the original if already valid.)
- **Escape newlines:** replace `\r\n`/`\n`/`\r` inside any field value with the literal two chars `\n`.
- **Guarantee all 15 keys** exist as strings; fill missing with `""`; trim whitespace.
- Return the clean `CrmLead`.

> The LLM should *also* be asked to do most of this (it's better at judgement calls like "which of these is the primary email"), but the validator is the **guarantee** — it must be correct even if the LLM did nothing.

## The API contract (full version in deep dive 01)
```
POST /api/import
  Content-Type: multipart/form-data
  body: file=<the .csv>        (field name: "file")
  → 200 { summary:{total,parsed,skipped}, records:[CrmLead...], skipped:[SkippedRow...] }
  → 400 { error: "..." }       (no file / empty / not CSV / too large)
  → 502 { error: "..." }       (LLM unavailable after retries and fallback also failed)

GET /api/import/stream        (Phase 4 bonus — SSE)
  emits: event:"progress" data:{done,total}   per batch
         event:"result"   data:{ImportResult} at the end
         event:"error"    data:{message}

GET /health → 200 { ok:true } (for uptime/keepalive; see deploy roadblocks)
```
`multer` memory storage; enforce `MAX_FILE_MB` (default 5). CORS restricted to the frontend origin (`CORS_ORIGIN`).

## The prompt (summary — full text and few-shot in `docs/PROMPT_ENGINEERING.md`)
- **System instruction:** "You are a precise data-extraction engine. Map each input CRM lead row to the fixed 15-field schema. Output JSON only, matching the provided schema. Never invent data; use `""` for anything absent. Rules: primary email/phone first, extras into `crm_note`; classify `crm_status` into exactly one allowed value or `""`; set `data_source` to an allowed value only when clearly referenced, else `""`; consolidate stray remarks/columns into `crm_note`."
- **Input:** the batch as a JSON array of `{rowIndex, cells:{<originalHeader>:<value>}}` so the model sees the **original column names** (that's the signal it maps from).
- **Structured output:** pass `responseSchema` = array of objects with the 15 fields, `crm_status`/`data_source` as `enum` types; `responseMimeType: "application/json"`.
- **Few-shot:** 2–3 worked examples covering the nasty cases (weird headers, multiple emails, free-text status → enum, real-estate possession). See PROMPT_ENGINEERING.md.
- **Determinism:** `temperature: 0` (or lowest allowed) for stable, repeatable mapping.

## PHASE 1 — Backend: CSV Parsing & `/api/import` skeleton *(one session)*
- **P1.1 — Upload + parse.** `POST /api/import` with Multer (memory), size/type guards, then `csvParser.ts` using PapaParse (`header:true`, `skipEmptyLines:true`, BOM handling, delimiter auto-detect). Produce `RawRow[]` with `cells` keyed by original headers (de-duplicate/normalize blank/duplicate header names). **Done when:** posting each of the 4 sample CSVs returns correct row/column counts; malformed/empty/non-CSV → clean `400`.
- **P1.2 — Passthrough result shape.** Wire the route to return an `ImportResult` where `records` is (for now) the raw rows shallow-mapped into the 15-field shape with no AI (obvious columns copied, rest blank), so the end-to-end contract is exercised before the LLM exists. **Done when:** the JSON contract in Part III is returned for all 4 samples; `summary` counts are correct.

**GATE 1:** the backend ingests any of the 4 sample CSVs and returns the well-formed `ImportResult` contract (pre-AI). Tag `gate-1-parsing`.

## PHASE 2 — Backend: Gemini Extraction Core *(one session — THE graded heart)*
- **P2.1 — LLM interface + Gemini provider.** `LlmProvider.ts` + `geminiProvider.ts` (the only file importing the Gemini SDK), using structured output (`responseSchema` + `responseMimeType`), `temperature:0`, and safety settings that don't block business data (see ROADBLOCKS §B). **Done when:** a hand-made 3-row batch returns 3 schema-valid candidates; no SDK import exists outside this file.
- **P2.2 — Prompt & schema.** Implement `prompts/extraction.ts` from `docs/PROMPT_ENGINEERING.md` (system instruction, JSON response schema with enums, few-shot). **Done when:** the messy sample rows map sensibly (weird headers resolved, multi-email split, status→enum).
- **P2.3 — Extractor (batch + concurrency + retry + fallback).** `extractor.ts` per Part III: chunking, promise-pool concurrency, exponential-backoff retry, and the deterministic fallback for permanently-failed batches. **Done when:** a forced API error on one batch does not fail the import (fallback kicks in, logged); large sample processes within rate limits.
- **P2.4 — Validator.** `validator.ts` implementing every rule in Part III §Validation, as pure functions. **Done when:** unit tests cover: skip rule, multi-email/phone overflow → `crm_note`, enum clamping (incl. fuzzy `"good lead"` → `GOOD_LEAD_FOLLOW_UP`), invalid `created_at` blanked, newline → `\n`, all-15-keys guarantee.
- **P2.5 — Wire the real pipeline.** Replace the P1.2 passthrough with parse → extract → validate → assemble. **Done when:** all 4 sample CSVs produce correct, clean, fully-populated results; the messy file's known-bad rows land in `skipped` with reasons; eyeball verification passes.

**GATE 2:** any of the 4 sample CSVs, posted to `/api/import`, returns correct AI-mapped, validated CRM records with an honest parsed/skipped split — proven by eyeball against the files. **This is the assignment's core; do not proceed until it genuinely works.** Tag `gate-2-extraction`.

## PHASE 3 — Frontend: the 4-Step Workflow *(one session)*
- **P3.1 — App shell + stepper + state machine.** `page.tsx` holds a 4-state machine (`upload → preview → confirm/loading → result`); `Stepper.tsx` shows progress. Tailwind base, responsive layout. **Done when:** the four states render and advance/reset correctly with mock data.
- **P3.2 — Upload step.** `UploadStep.tsx`: drag-and-drop **and** file picker; validate `.csv`, non-empty; friendly errors. **Done when:** both upload methods work; bad files rejected with a message.
- **P3.3 — Preview step (no AI).** Client-side PapaParse (`lib/csv.ts`) → `PreviewTable.tsx` via the reusable `DataTable.tsx`: **sticky header, horizontal + vertical scroll**, responsive, row/column counts. **Done when:** a wide CSV scrolls both ways with the header pinned; no backend call yet.
- **P3.4 — Confirm + call backend.** On Confirm, POST the file to `/api/import` (`lib/api.ts`); show a loading state. **Done when:** clicking Confirm returns real results from the Phase-2 backend.
- **P3.5 — Results step.** `ResultView.tsx`: parsed records table + skipped records (table or tab) + totals; "Import another file" reset. **Done when:** parsed/skipped/totals display correctly for all 4 samples end-to-end.

**GATE 3:** full local end-to-end — upload a sample CSV, preview it, confirm, and see correct AI-parsed results, all in the browser against the local backend. Tag `gate-3-frontend`.

## PHASE 4 — UX, Errors, Progress & Dark Mode *(one session)*
- **P4.1 — Loading & progress.** Replace the plain spinner with real progress: either the SSE `GET /api/import/stream` (per-batch `done/total`) with a progress bar, or at minimum a determinate-feel staged loader. **Done when:** the user sees progress advance during a large-file import (graded: loading states).
- **P4.2 — Error handling everywhere.** Network/backend/timeout errors surface as friendly, actionable messages with a retry action; the app never white-screens. **Done when:** killing the backend mid-import shows a clean error + retry, not a crash.
- **P4.3 — Responsive + dark mode.** Verify mobile/tablet/desktop; add a dark-mode toggle (Tailwind `dark:`). Tables scroll within their own container; the page body never scrolls horizontally. **Done when:** looks correct at 375px / 768px / 1280px in both themes.
- **P4.4 — Empty/edge states.** Empty results, all-skipped file, huge file — all render sensibly. **Done when:** each edge case has a designed state, not a blank screen.

**GATE 4:** the app is production-feeling — progress, error recovery, responsive, dark mode, edge states. Tag `gate-4-ux`.
